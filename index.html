1. 동시 예약 요청으로 인한 중복 예약 발생 (Race Condition)
개선 전 흐름 (Race Condition 발생) — 시퀀스 다이어그램
개선 후 흐름 (트랜잭션으로 차단) — 시퀀스 다이어그램
🔴 문제 원인
  • 티켓 예매/공간 대여 도메인에서 동시 요청 시 같은 좌석·시간대에 중복 예약이 발생했습니다.
  • find() → 확인 → save() 사이의 간격에서 여러 요청이 동시에 통과하는 Race Condition이 원인이었으며, seatSchema에 Unique Index가 없어 DB 레벨에서도 차단이 불가능한 구조였습니다.
  • k6 부하테스트(동시 200명)로 재현한 결과 중복 예약 9건이 발생했으며, 같은 좌석에 여러 명이 입장하는 심각한 비즈니스 문제로 이어질 수 있었습니다.
🟢 해결 과정
  • DB 레벨 방어로 seatSchema에 { showId, date, time, seatNumbers } 복합 Unique Index를 추가했습니다.
  • 애플리케이션 레벨 방어로 mongoose.startSession()으로 트랜잭션을 시작하고 find()와 save() 모두 동일 세션 안에서 처리했습니다.
  • 트랜잭션 충돌 시 발생하는 WriteConflict(code: 112)를 catch에서 잡아 500 대신 400으로 반환해 클라이언트에 명확한 오류 메시지를 전달했습니다.
session.startTransaction() .session(session) commitTransaction abortTransaction error.code === 11000 WriteConflict
🟡 결과
  • k6 부하테스트(동시 200명) 기준, 중복 예약이 9건 → 0건으로 완전히 해결됐습니다.
  • 트랜잭션 적용으로 평균 응답시간이 315ms → 1,100ms로 증가했지만, 티켓 예매 도메인 특성상 성능보다 데이터 정합성을 우선해 트레이드오프를 감수했습니다.
  • 좌석 예약뿐 아니라 공간 대여 도메인에도 동일하게 적용해 서비스 전반의 중복 예약 문제를 해결했습니다.

2. Seat/Rental 컬렉션 분리로 인한 데이터 불일치
트랜잭션 원자성 보장 — 시퀀스 다이어그램
🔴 문제 원인
  • 좌석 예약(Seat)과 공간 대여(Rental)는 각각 별도 컬렉션에 저장되는 구조였습니다.
  • Seat 저장 성공 후 Rental 저장 단계에서 오류가 발생하면, Seat 데이터만 남고 Rental은 저장되지 않는 부분 저장 상태가 발생했습니다.
  • 예약은 완료됐지만 대여 정보가 없는 데이터 불일치가 생겨, 관리자가 수동으로 정합성을 맞춰야 하는 운영 리스크가 있었습니다.
🟢 해결 과정
  • mongoose.startSession()으로 트랜잭션 세션을 시작하고, Seat와 Rental 저장 모두 동일 세션 안에서 처리했습니다.
  • 두 저장 작업을 하나의 트랜잭션으로 묶어 어느 한 쪽이 실패하면 abortTransaction()으로 전체를 롤백하도록 구성했습니다.
  • Race Condition 해결에서 적용한 트랜잭션 구조를 공간 대여 도메인에 동일하게 확장 적용해 서비스 전반의 데이터 정합성을 보장했습니다.
mongoose.startSession() .session(session) commitTransaction abortTransaction withTransaction
🟡 결과
  • Seat 저장 성공 + Rental 저장 실패 시나리오에서 Seat가 자동 롤백되어 부분 저장 케이스가 완전히 제거됐습니다.
  • 수동 정합성 작업 없이 예약-대여 데이터가 항상 일치하는 구조가 됐습니다.
  • Race Condition 해결(DB 레벨 Unique Index) → 트랜잭션 원자성 보장(애플리케이션 레벨)으로 이어지는 2단계 데이터 정합성 방어 구조를 완성했습니다.
3. 결제 금액 클라이언트 위변조 가능성
결제 검증 흐름 — 시퀀스 다이어그램
🔴 문제 원인
  • 프론트엔드에서 Toss 결제 완료 후 paymentKey를 generateRandomString()으로 직접 생성해 백엔드로 전달하는 구조였습니다.
  • 백엔드가 Toss 서버에 실제 결제 검증을 요청하지 않고 클라이언트가 보낸 값을 그대로 신뢰해 DB에 저장했습니다.
  • 브라우저 개발자 도구로 amount 값을 조작한 요청이 그대로 통과할 수 있는 결제 위변조 리스크가 존재했습니다.
🟢 해결 과정
  • requestPayment() 이후 handlePaymentSuccess()를 직접 호출하던 구조를 제거하고, Toss가 실제 paymentKey를 successUrl에 자동으로 부착해 리다이렉트하는 흐름으로 수정했습니다.
  • 백엔드에서 Toss Payments 서버사이드 검증 API(/v1/payments/confirm)를 직접 호출해 실제 결제금액과 요청금액의 일치 여부를 확인했습니다.
  • 검증 통과 후에만 DB에 결제 정보를 저장하는 순서를 보장했으며, 티켓/공간/경매/굿즈 4개 도메인 결제 전체에 동일하게 적용했습니다.
/v1/payments/confirm paymentKey orderId amount 검증 서버사이드 검증 successUrl 리다이렉트
🟡 결과
  • Postman으로 금액 조작 요청(실제 결제금액 340,000원 → 요청 1,000원)을 직접 재현한 결과, Toss 서버사이드 검증 단계에서 FORBIDDEN_REQUEST(403) 반환으로 차단됐습니다.
  • 클라이언트에서 금액을 조작한 요청이 DB에 저장되지 않고 서버 레벨에서 완전히 차단되는 구조가 확인됐습니다.
  • 티켓/공간/경매/굿즈 4개 도메인 결제 모두 동일한 검증 흐름을 적용해 서비스 전반의 결제 보안을 완성했습니다.

5개월간의 LangChain/RAG/NLP 이론 학습을 바탕으로, 9개 챗봇 프로젝트 중 프라이버시와 성능 최적화에 집중한 PrivateGPT를 포트폴리오 대표 프로젝트로 선정했습니다.
서비스 소개

업로드한 문서에 대해 질문할 수 있는 로컬 기반 RAG 챗봇. 임베딩 모델과 LLM 모두 로컬(Ollama + Mistral)에서 실행되어 외부 API 호출 없이 동작하며, 민감한 문서도 안전하게 처리할 수 있는 프라이빗 AI 시스템.

OpenAI API를 사용하면 문서 데이터가 외부 서버로 전송되어 프라이버시 문제가 발생하고 API 호출 비용도 발생합니다. 이를 해결하기 위해 임베딩과 LLM을 모두 로컬에서 실행하는 Ollama + Mistral 조합을 선택했습니다. "Private"GPT라는 이름처럼, 데이터가 외부로 나가지 않는 구조가 이 프로젝트의 핵심 설계 의도입니다.

담당 역할

RAG 파이프라인 설계 (문서 로딩 → 청킹 → 임베딩 → 벡터 검색 → 답변 생성), 임베딩 캐싱 구조 구현, 대화 히스토리 체인 연결, Streamlit UI 구현

🔴 문제 원인
  • Streamlit UI에는 이전 대화가 표시되지만, 실제 RAG 체인은 매번 현재 질문만 단독으로 처리했습니다.
  • "방금 말한 거 더 자세히 설명해줘" 같은 맥락 의존 질문이 들어오면, retriever가 "방금 말한 거"를 독립적인 검색 쿼리로 처리해 관련 없는 문서를 검색했습니다.

  • 재구성된 질문으로 retriever를 invoke한 뒤, 원래 질문 + 대화 히스토리 + 검색된 context를 함께 최종 LLM에 전달해 맥락 인식 답변을 생성합니다.

3. 관련성 없는 문서도 무조건 검색 결과에 포함되는 문제
🔴 문제 원인
  • 기본 as_retriever() 설정은 관련성 점수와 무관하게 무조건 상위 k개 문서를 반환합니다.
  • 문서와 전혀 관련 없는 질문("오늘 저녁 뭐 먹을까?")에도 가장 유사한 청크 4개를 반환해, LLM이 엉뚱한 컨텍스트로 답변을 생성하거나 할루시네이션이 발생했습니다.
🟢 해결 과정
  • search_type="similarity_score_threshold"를 적용해 관련성 점수가 기준값 이하인 문서는 검색 결과에서 제외했습니다.
  • 로컬 모델(Mistral) 임베딩은 OpenAI보다 점수 범위가 낮게 나오는 특성을 고려해 threshold를 0.05로 설정했습니다.
# 개선 전: 관련성 무관하게 무조건 4개 반환
retriever = vectorstore.as_retriever()

# 개선 후: 관련성 점수 기준 이하 문서 제외
retriever = vectorstore.as_retriever(
    search_type="similarity_score_threshold",
    search_kwargs={"score_threshold": 0.05, "k": 4}
)

🟡 결과
  • 문서와 무관한 질문("오늘 저녁 뭐 먹을까?")에 "모른다"고 정확히 응답하는 것을 확인했습니다.
  • 관련 없는 컨텍스트가 LLM에 전달되지 않아 할루시네이션 발생 가능성을 낮췄습니다.
  • 로컬 모델의 점수 특성에 맞게 threshold를 조정하는 과정에서 임베딩 유사도 점수 구조를 직접 분석했습니다.

1. 초기 배포 성공률 약 70% — 오류 원인이 매번 달라지는 CI/CD 불안정
CI/CD 파이프라인 (GitHub → Docker Hub → AWS EC2) — 아키텍처 다이어그램
🔴 문제 원인
  • 공모전 일정에 맞춰 기능을 반복 배포해야 했으나 GitHub Actions 기록 기준 총 86회 중 26회 배포가 실패했습니다 (성공률 약 70%).
  • Secrets 설정 오류(이메일 주소 입력), Dockerfile 파일명 오타(Dokcerfile), 보안 그룹 SSH 포트 차단, Docker 권한 미설정 등 오류 원인이 매번 달라 장애 감지와 대응에 수 분이 걸렸습니다.
  • Anthropic SDK는 ANTHROPIC_API_KEY를 자동 인식하는데 AI_API_KEY로 설정해 AI 기능 전체가 동작하지 않는 문제도 발생했습니다.
🟢 해결 과정
  • GitHub Actions 워크플로우를 직접 설계해 main 브랜치 push 시 Docker 이미지 빌드 → Docker Hub push → EC2 pull & 컨테이너 교체까지 전 과정을 자동화했습니다.
  • --no-cache 옵션과 fetch-depth: 0을 표준화해 캐시 충돌과 코드 미반영 문제를 원천 차단했습니다.
  • 9개 민감 정보를 GitHub Secrets로 관리하고, ANTHROPIC_API_KEY는 Dockerfile ARG에서 제외해 이미지 레이어에 키가 남지 않도록 런타임 전용 주입(docker run -e)으로 설계했습니다.
  • docker stop sensitive || true 패턴으로 컨테이너 미존재 시에도 파이프라인이 중단되지 않도록 방어했습니다.

2. 배포 exit 0인데 서비스는 구버전 — Silent Failure 디버깅
EBS 볼륨 확장 전 / 후 (EC2 인스턴스) — 구성도
🔴 문제 원인
  • 파이프라인은 정상 완료(exit 0)되는데 실제 서비스는 계속 구버전 코드로 동작했습니다.
  • 에러 로그 없이 구버전이 유지되는 Silent Failure였기 때문에 원인 파악에 시간이 걸렸습니다.
  • docker pull 실행 시 no space left on device 에러가 조용히 발생하고 있었고, docker images 명령으로 확인하니 반복 배포로 누적된 이미지 4개(각 ~1GB)가 7.6GB EBS 볼륨을 가득 채운 상태였습니다.
  • 새 이미지를 pull할 공간이 없어 EC2가 기존 이미지를 그대로 재사용하고 있었던 것이었습니다.
🟢 해결 과정
  • docker images 명령으로 누적 이미지 목록을 확인해 원인을 특정했습니다.
  • AWS 콘솔에서 EBS 볼륨을 7.6GB → 30GB로 확장했습니다.
  • growpart /dev/nvme0n1 1 + resize2fs /dev/nvme0n1p1로 파티션·파일시스템을 무중단 리사이즈했습니다.
  • 불필요한 기존 이미지를 삭제(docker image prune)해 공간을 확보하고, 워크플로우에 --no-cache 옵션을 표준화해 레이어 누적 자체를 방지하는 구조를 잡았습니다.

🟡 결과
  • docker pull이 정상화되어 이후 배포 = 서비스 반영이 일치하게 됐습니다.
  • docker ps, docker logs를 활용한 배포 후 검증 절차를 정립해 동일 유형의 Silent Failure 재발을 방지했습니다.
